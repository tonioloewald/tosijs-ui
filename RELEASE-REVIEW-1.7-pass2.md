## Blockers (must fix before tagging)

### B1 — A breaking release is numbered as a minor (`CHANGELOG.md:27`)

`^1.6.x` resolves `1.7.0`, so a consumer who uses nothing but `<tosi-rating>` gets, on an unrelated `bun install`: a different editor engine, `theme`/`options`/`ace` removed, **`editor` silently retyped in place** (ACE `Editor` → CM `EditorView`, so `el.editor.session.*` is a runtime `TypeError` — no shim covers this), 23 new mandatory runtime packages (the library previously had **zero** `dependencies`), and a 3.2× IIFE for CDN `<script>` consumers on a floating URL. The CHANGELOG frames the auto-upgrade as a fact of life — that is the trap, not a feature.

**Fix:** ship as **2.0.0**, and rewrite the closing line of the Breaking section to say `^1.6` consumers stay on 1.6.x and opt in deliberately. The window is open and costless _today_ — `package.json` has not been cut yet. Once `1.7.0` is on npm there is no clean remedy.

### B2 — Version never bumped (`package.json:3`)

`package.json` = `1.6.22`; `src/version.ts`, `dist/version.js`, `llms.txt` (`# tosijs-ui v1.6.22`) and the pre-rendered `docs/` all agree — while `CHANGELOG.md` has a full `## 1.7.0` section and `src/code-editor.ts` warns "removed in tosijs-ui 1.7". There is **no** `prepublishOnly`/`prepack` hook, so `npm version && npm publish` would ship a tarball whose runtime `version` export and `llms.txt` header lie.

**Fix:** the project's own documented order — bump `package.json` → `bun run build` (restamps `version.ts`, `dist/`, `llms.txt`, `docs/`, ePub) → commit the regenerated artifacts. Per B1 the number is `2.0.0`.

## Majors (fix now — correctness first)

**M1 — `<tosi-code>`'s lazy CM load has no connect-state guard** (`src/code-editor.ts:286`).
`connectedCallback` fires `import('./code-editor-cm').then(…)` and the `.then` unconditionally creates the editor; `disconnectedCallback` clears `_loadPromise`, so a reconnect starts a _second_ load. Reproduced: `append → remove → append` before the chunk resolves leaves **two** `.cm-editor` views in the host, with `_handle` pointing only at the second — the first `EditorView` plus its `watchDarkMode` listener is retained forever in the module-level `darkListeners` set. `append → remove` builds an `EditorView` into a detached shadow root that is never destroyed. This is the very leak M1 set out to fix, relocated into the pre-load window (which is wide on the cold chunk fetch — doc-browser nav and `closeEditor()` both remove editors mid-flight).
_Fix:_ capture a token before the import; in the `.then`, bail (and destroy the fresh handle) if the token is stale or `!this.isConnected`. Add tests for `append→remove→append` and `append→remove` **without** awaiting the mount — every current test awaits it, which is exactly why this slipped through.

**M2 — `showDiff()` before hydration permanently bricks the editor** (`src/code-editor.ts:174`).
`showDiff()` starts `if (!this.partsReady) return` and nothing re-applies the request. Worse than filed: `partsReady` reads `this.parts.diffHost`, and tosijs's `parts` getter **caches its query root on first access** — so touching it pre-connect permanently roots the proxy at the light-DOM element, and every later `this.parts.X` throws. `createCmEditor(this.parts.host, …)` inside the load `.then()` then throws, and **CodeMirror never mounts at all**. Merely reading `el.showingDiff` before insertion is enough. External consumers only; silent and unrecoverable.
_Fix:_ gate on `this.shadowRoot`/hydration state without touching `parts`, and stash a pending `showDiff` request to apply after hydration.

**M3 — The scope-capture epilogue can emit a `SyntaxError` that kills the whole live example** (`src/live-example/code-transform.ts:199`).
`extractTopLevelBindingNames` is string/comment-unaware, so a line at column 0 inside a template literal (`const in a template is just prose`) yields the name `in`, and `buildScopeCapture` emits `try{__tosiScope.in=in}catch(_e){}` — a **parse** error no `try/catch` can absorb. The `AsyncFunction` never compiles and the example dies with "example loads without error". The code comment claiming an over-match "degrades completions rather than breaking the example" is false. Zero current-corpus impact, but it is a live trap for doc authors and for users editing an example in place.
_Fix:_ filter extracted names to `/^[A-Za-z_$][\w$]*$/` minus the reserved-word set before emitting the epilogue; wrap epilogue generation so a failed compile falls back to no capture.

**M4 — The hand-rolled declaration scanner silently drops every binding after one unbalanced bracket** (`src/live-example/code-transform.ts:167`).
`indexAtDepth0`/`readDeclaration`/`patternNames` count brackets over raw text with no lexer. `const open = '('` → depth never returns to 0 → the rest of the file is consumed and `declRe.lastIndex` skips past it. Triggers are mundane: `s.split(')')`, `list.join('{')`, `// :-)`, `/[(]/`. Verified end-to-end: one line `const label = 'Items ('` silently dropped `count`, `addItem`, and `Widget` from the captured scope — so the headline B2 runtime-value autocomplete yields **zero completions**, with no error, no warning, and no test lane catching it (the 10 existing tests contain not one string, comment, or regex). This is the second round of the same bug class (commit `3062395f` already fixed 12/26 missed bindings on this release's own SIMD demo) — the signature of reimplementing a parser.
_Fix:_ use a real parser. `acorn` already resolves in `node_modules` today (via tjs-lang), or lazily reuse the lezer parser that ships with `@codemirror/lang-javascript`. If neither lands, at minimum fail _safe_: on non-zero depth at EOF, resume scanning at the next line instead of swallowing the tail.

**M5 — The doc-system hydration bundle grew 3.2× and blocks first paint** (`src/doc-system/site/orchestrator.ts:268`).
Measured from the committed artifacts: `dist/iife.js` 418KB → 1,198KB raw, **121,240 → 386,687 bytes gzipped**. `generate-site.ts:271` emits it as a plain blocking `<script src="/iife.js">` on every generated page, and generated pages set `body{opacity:0}` until hydration — so 387KB gz + ~1.2MB of parse now gates **first visible text**, not just interactivity, on every page of every site built with `tosijs-ui/site`, whether or not a reader ever opens an editor. The `import('./code-editor-cm')` boundary is a real lazy chunk for ESM consumers, but `--format=iife` cannot split.
_Note:_ the CHANGELOG calls this deliberate ("the in-page editor is the point of the doc-system"), which is a defensible product call — but nothing mitigates it. _Fix (or accept explicitly):_ emit the hydration bundle as ESM with `--splitting` + `<script type="module">` so the existing boundary becomes a real chunk (`bundle-guard.classicScriptSyntaxError()` would need to be module-aware); keep the IIFE strictly for the CDN path. If that's too big for this release, at minimum add `defer`.

**M6 — The WASM SIMD demo burns a CPU core forever on a docs page** (`src/live-example/component.ts:268`).
Measured on the built `/component/` page in Chrome, demo scrolled off-screen, no input: `TaskDuration` +0.452s per 4s wall clock (≈7% of a core marginal, ≈97% of all script execution on the otherwise-idle page). The rAF loop's only exit is `!document.body.contains(canvas)` — no `IntersectionObserver`, no `document.hidden` check — and the `idle > 45` branch drives a Lissajous curve so it never quiesces without input. Each frame: 100k particle updates + a 576KB `ImageData` fill + 100k stores + `putImageData`, over 2.4MB of Float32Arrays; load pays a 300×100k warm-up.
_Fix:_ `IntersectionObserver` to pause off-screen, bail on `document.hidden`, honor `prefers-reduced-motion`. A few lines.

**M7 — The Playwright lane tests a haltija-polluted DOM and poisons the doc-test lane** (`playwright.config.ts:79`).
The new `webServer` block runs `bun start` with no `env`, and `tosijs-site.config.ts:55` sets `haltijaDev: true` — so every local Playwright page (chromium/firefox/webkit) loads the haltija dev overlay while CI's chromium (with `CI=true`) does not. The two runs assert against **different DOM**, and tests are already being contorted around it (`localization.pw.ts` dodges "a dev overlay's 'Filter events…' control"). Second symptom, reproduced: those pages register as haltija windows, and `bin/dev.ts --test` reuses any haltija with `windows.length > 0` — with stale windows present `bun run test-browser` failed outright (`Browser tests timed out`, exit 1); after `pkill -f haltija` the same command passed 33/33. A green codebase yields a red doc-test lane in the middle of the release gate.
_Fix:_ add `env: { HALTIJA_DEV: '0' }` to the `webServer` block, and make the test-browser harness validate the reused haltija is live (or just pass `-f`). Note `reuseExistingServer: !process.env.CI` still adopts a developer's already-running overlay-injecting `bun start`, so the env fix alone isn't sufficient.

**M8 — Agent docs contradict the config this release changed** (`CLAUDE.md:34`, `:36`, `:45`, `:54`, `:68`, Publishing step 2; also `tests/code-editor.pw.ts:7`).
This branch added a real `webServer` block _and_ a CI `e2e` job — and CLAUDE.md still says, in seven places, that Playwright does not auto-start the server and that CI runs unit tests only. (Chronology: commit `a5164a32`, "correct CLAUDE.md drift found by the release review", _was_ true when written; the later `2a36a63f` invalidated it and didn't touch the docs.) An agent triaging a red `e2e` job reads "no Playwright tests in CI" and concludes the job can't exist; one debugging a Playwright failure burns budget on "is the server up?". Second consecutive release where the lane docs and the lane config disagree.
_Fix:_ update all seven spots (delete the now-obsolete "Caveat" paragraph rather than patching it) and drop the stale header sentence in `tests/code-editor.pw.ts`.
